# 第一章

# 什么是软件文档

  1. 软件文档描述系统功能，刻画子系统间的相互关系，是为开发者提供的精确、完整的指导资料。软件文档是软件开发者之间的沟通渠道，是具体工作的安排表，是系统的开发标准。

# 软件文档的作用

  1. 软件文档的本质作用是桥梁、是纽带，连接着软件开发方、管理人员，用户以及计算机，将其构成一个相互影响、相互作用的整体。

# 软件文档的分类及作用

  • 可行性研究报告：说明软件开发项目的实现在技术，经济和社会条件方面的可行性，评估为了合理的达到开发目标而可能选择的各种方案
  • 项目建议书：为软件项目实施方案制定出具体计划
  • 招投标文件
  • 需求分析书：开发人员将客户需求转化为文字记录下来
  • 概要设计书：从需求设计书出发，抽象出系统的功能模块，数据库要求，体系结构等大方向的问题.
  • 详细设计书：详细设计书以概要设计书为基础，对已拆分出来的子系统的功能和模块逐个进行设计
  • 项目验收总报告：包括对所完成系统的测试，验收和总结，为了在产品交付前尽可能多的发现问题并及时修正问题

# 一个软件项目从立项到结尾共有几个阶段：

  1. 立项，投标，需求分析，概要设计，详细设计，软件编码，软件测试，维护，验收

# 名词解析 —— 体系结构，功能模块

  • 体系结构：指的是软件系统的结构，它描述了软件系统中各个组件之间的关系以及它们是如何协同工作的。体系结构主要关注的是系统整体的结构，例如分层、客户端 - 服务器等。体系结构设计的好坏对软件的性能、可扩展性、可维护性等有很大的影响。

  • 功能模块：功能模块是指软件系统中实现特定功能的、相互独立的模块。在软件开发中，通常将系统按照不同的功能模块进行划分，每个模块完成一个具体的功能，同时与其他模块相互协作，共同实现系统的整体功能。

    常见的功能模块包括数据存储模块、用户管理模块、权限管理模块、日志记录模块、

# 第三章 —— 软件项目立项阶段文档写作案例分析

# 名词解析 —— 市场风险，技术风险

  • 市场风险：指由于市场因素引起的商业风险，包括市场需求、竞争、价格波动等因素。例如，当某种产品或服务的市场需求下降时，销售量和收入都可能受到影响，从而导致经济损失。
  • 技术风险：指由于技术因素引起的商业风险，包括技术研发、产品设计、制造和运营等方面。例如，当某种新技术的应用不成熟或存在缺陷时，产品的质量和性能可能无法满足市场需求，从而导致企业的经济损失。

# 第四章 —— 需求分析书案例分析

# 需求分析书是干什么的

  • 一般是对某个市场或者是客户群来讲的，类似于调研报告，重点是体现出产品要满足哪些功能，哪些是重点、热点。

# 需求分析书有哪些功能

需求分析书的主要功能是确保软件开发满足用户需求，有效地指导软件开发过程，规范软件开发流程，提高软件开发效率和质量。

# 名词解析 —— 用例方法，用例规约，IPO 图，响应时间，故障处理

  • 用例方法：是一种软件开发过程中，用于描述系统如何满足用户需求的技术。它是一种面向用户需求的方法，通过识别系统的各种用例来描述系统的功能需求
  • 用例规约：是用例方法中的重要组成部分，用于详细描述每个用例的业务流程、前置条件、后置条件、基本流程和扩展流程等内容。
  • IPO 图：是一种描述系统输入、处理和输出的图形化工具，它由三个部分组成：输入、处理和输出，以显示系统执行的功能和数据流。
  • 响应时间：指应用执行一个操作所需的时间，包括从发出请求开始到最后收到响应所需要的时间。
  • 故障处理：）是指在软件系统运行过程中，当出现故障时系统能够自动或手动进行处理的能力。它通常包括故障诊断、故障处理和故障恢复等过程。

# 第五章 —— 需求分析书案例分析二

# 什么是功能需求，包括哪些

功能需求是指系统或软件需要实现的特定功能或行为。这些功能需求通常来自于用户需求或业务需求，并且是系统或软件必须实现的核心部分。

常见的功能需求包括：

  1. 用户登录和注册：系统需要提供用户登录和注册功能，包括身份验证、密码重置等。
  2. 数据录入和管理：系统需要支持数据的录入、存储、查询、修改、删除等操作。
  3. 数据分析和报告：系统需要支持对数据进行分析和生成相应的报告。
  4. 安全控制：系统需要提供安全控制功能，包括数据的访问控制、用户权限管理等。

# 非功能需求包括哪些

  • 性能需求 —— 用户在软件响应速度、结果精度、运行时资源消耗量等方面的要求
  • 可靠性要求 —— 用户在软件失效的频率、严重程度、易恢复性，以及故障可预测性等方面的要求。
  • 易用性要求 —— 用户在界面的易用性、美观性，以及对面向用户的文档和培训资料等方面的要求。
  • 运行环境约束 —— 用户对软件系统运行环境的要求。
  • 外部接口需求 —— 用户对待开发软件系统与其他软件系统或硬件设备之间的接口的要求。
  • 安全性需求 —— 用户在身份认证、授权控制、私密性等方面的要求。
  • 可保障性要求 —— 用户在软件可配置性、可扩展性、可维护性、可移植性等方面的要求。
  • 输入输入需求
  • 数据管理能力需求
  • 故障处理需求 —— 一旦运行发生异常应当有信息提示，用户无法进行违规操作

# 第六章 —— 需求分析书案例三

# 建设原则有哪些

  • 先进性
  • 实用性
  • 高可靠性
  • 开放性
  • 可维护性
  • 可移植性
  • 扩展性

# 名词解释 —— 可移植性，可维护性，扩展性

chatgpt 给出的回答

  • 可移植性 —— 是指软件在不同硬件、操作系统或环境中，能够被方便地转移或迁移的能力。软件的可移植性越高，说明该软件能够适应多种不同的硬件和软件环境，提高了软件的可用性和灵活性。
  • 可维护性 —— 是指软件在发布后，需要对软件进行维护、修改和更新时，容易被开发人员进行修改的能力。具有良好的可维护性的软件能够让开发人员更加轻松地进行修改、更新和修复，从而节省时间和精力。
  • 扩展性 —— 是指软件在面对新的需求时，可以通过扩展软件的功能和性能来满足这些新的需求的能力。具有良好扩展性的软件能够有效地满足不断变化的需求，具有更长久的生命周期。

# 第七章 —— 概要设计书案例分析一

# 概要设计是干什么的

  • 概要设计是软件开发中非常重要的一个环节，它决定了软件的整体结构，决定了整个开发过程人员、时间的安排。

# 概要设计完成什么样的目标

  • 在概要设计阶段，需要将软件系统需求转化为未来系统的设计，逐步开发强壮的系统构架，将系统进行合理的子系统，功能的分解

# 概要设计有个什么样的结果

  • 概要设计的结果通常是一个设计文档，它描述了系统的整体结构、各个模块之间的关系、数据结构、算法、接口定义等，为详细设计提供了一个框架。

# 问答题 —— 健壮性，系统划分的基本原则是什么

  • 健壮性 —— 健壮性指的是系统对于非法输入、意外情况、异常情况的处理能力。
  • 系统划分的基本原则是什么 —— 按照业务闭环进行系统拆分／组织架构划分，实现闭环／高内聚／低耦合，减少沟通成本
  • chatgpt 回答系统划分的基本原则是将系统分成若干个模块，每个模块负责不同的功能，模块之间尽可能独立，各模块之间通过接口交互。

# 第九章 —— 概要设计书案例分析三

# 名词解释 —— 软件生命周期，瀑布模型，螺旋模型

  • 软件生命周期 —— 软件生命周期 (Software Life Cycle,SLC) 是软件的产生直到报废或停止使用的生命周期。. 软件生命周期内有问题定义、 可行性分析 、总体描述、 系统设计 、编码、调试和测试、验收与运行、维护升级到废弃等阶段，也有将以上阶段的活动组合在内的迭代阶段，即迭代作为生命周期的阶段。
  • 瀑布模型 —— 瀑布模型是将软件生存周期的各项活动规定为按固定顺序而连接的若干阶段工作，形如瀑布流水，最终得到软件产品。
  • 螺旋模型 —— 螺旋模型是一种演化软件开发过程模型，它兼顾了快速原型的迭代特征以及瀑布模型的系统化与严格监控。

# 第十章 —— 详细设计书案例分析一

# 名词解释 —— 算法，N-S 图，PAD 图

  • 算法 —— 算法是指用来解决问题的一系列清晰而确切的规程。在计算机科学中，算法是指解决问题的步骤和规则，通常被表示为一个有限的指令序列，其中每个指令表示一个或多个操作。
  • N-S 图 ——N-S 图（Nassi-Shneiderman Diagram）是一种用于表达结构化程序设计概念的图形化表示法。它可以帮助程序员理解程序的流程控制结构，是一种类似于流程图的图形化表示方法。
  • PAD 图 ——PAD（Program Analysis and Design）图是一种结构化分析和设计方法中的图形化工具，用于描述程序的流程控制结构。它通常由框、箭头和标签等基本符号构成，用于表示程序的模块、数据流和处理流程等。PAD 图可以帮助程序员更好地理解程序的结构和流程，从而更好地进行程序设计和开发。

# 问答题

  • 详细设计是什么

    在概要设计完成整个系统的宏观设计后，再由详细设计对概要设计中提出的各个功能模块逐个进行详细分析，全面展开。

  • 详细设计要详细到什么程度，这样做的目的是什么？

    详细设计要包含所有模块的各方各面的内容，要细致到在编码之前没有任何不确定因素，包括函数名，类名等的命名，各个页面表示项目的属性，长度，特殊的限制，具体的算法也要在此阶段确定。

    目的是为了在编码阶段能将设计结果无歧义的直接转化为某种开发语言；另一方面也是为了以后维护人员在代码维护时能提供实用的帮助，不用在纷繁众多的代码中找不到切入点。

    chatgpt 的回答

    详细设计需要详细到足以让程序员根据设计图来编写代码的程度。具体来说，它应该包括所有必需的类、方法、变量、算法等实现细节，以及各种数据结构、算法和接口之间的具体实现方法。此外，详细设计应该定义和描述实现细节，以确保所有开发人员在编写代码时遵循一致的标准和方法。

    详细设计的目的是为了让开发人员更容易地实现系统，并减少系统实现的错误和问题。通过在详细设计阶段明确定义和描述所有的实现细节，可以让开发人员更容易地理解系统的要求和目标，并且可以更快地编写正确的代码。此外，详细设计还可以帮助团队成员更好地协作，减少开发成本，加快开发进度，提高软件质量。

  • 通常可以借助哪些形式来描写算法？

    1. 自然语言：通过文字、语言描述算法思路、步骤和实现过程。
    2. 流程图：使用流程图来描述算法的流程和步骤。
    3. 伪代码：用一种类似编程语言的语法来描述算法，但不必遵循具体编程语言的语法规则，强调算法的逻辑结构。
    4. 实现代码：使用具体编程语言来实现算法。

# 名词解释 —— 判定表，判定树

  • 判定表 —— 是一种使用表格形式描述程序行为的技术。它用来描述程序输入和输出之间的关系，并且可以使用不同的输入和条件的组合来产生不同的输出。在判定表中，每个行表示一种情况，每个列代表一个条件，每个单元格代表一个行为。判定表可以用来识别程序中的逻辑错误和缺陷，特别适用于测试决策逻辑比较复杂的程序。
  • 判定树 —— 判定树（Decision Tree）是一种使用树形结构表示程序行为的技术。它用来描述程序输入和输出之间的关系，并且可以使用不同的输入和条件的组合来产生不同的输出。判定树以一棵树的形式表示程序的决策过程，每个节点代表一个条件，每个分支代表一个可能的结果。判定树可以用来设计测试用例，特别适用于测试决策逻辑比较简单的程序。

# 第十二章 —— 软件项目结束阶段文档写作案例分析

# 什么是功能测试

功能测试是一种软件测试方法，旨在测试软件系统是否按照其规格说明书和设计规范的要求正常工作。功能测试的重点在于测试软件的各个功能模块，以确保它们的行为符合预期。

在功能测试中，测试人员通过输入一系列测试用例，测试软件在各种输入情况下是否能够正确地响应，并且是否能够产生正确的输出结果。这些测试用例通常根据规格说明书、用户手册和其他软件设计文档中的要求编写。

功能测试可以帮助开发人员发现和解决软件系统中的各种错误，包括逻辑错误、用户界面问题、性能问题和兼容性问题等。

# 名词解释 —— 黑盒测试，白盒测试

  • 黑盒测试 —— 黑盒测试是指测试人员在不考虑程序内部结构和实现细节的情况下，只根据软件需求规格说明书，对软件的功能、接口、性能等进行测试的方法。黑盒测试通常从用户的角度出发，着重测试软件的功能是否符合需求，输入和输出是否正确，界面是否友好等。
  • 白盒测试 —— 白盒测试是指测试人员基于程序内部结构和实现细节，对软件进行测试的方法。白盒测试通常从开发人员的角度出发，通过对程序的代码、逻辑、数据流等进行分析，设计测试用例，测试程序是否能按照预期的方式工作，包括是否能正确处理各种输入和异常情况，是否能达到设计要求的性能和可靠性等。

# 问答题

  1. 为什么要进行测试？

    1、发现被测对象与用户需求之间的差异，即缺陷。

    2、通过测试活动发现并解决缺陷，增加人们对软件质量的信心。

    3、通过测试活动了解被测对象的质量状况，为决策提供数据依据。

    4、通过测试活动积累经验，预防缺陷出现，降低产品失败风险。
